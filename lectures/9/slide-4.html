Adversarial Attacks

🎯
Ухилення (Evasion)
Мінімальна зміна input для обходу моделі. Fraud, що мімікрує легітимну
☠️
Отруєння (Poisoning)
Шкідливі дані у навчальному датасеті → backdoor у моделі
🔍
Вилучення моделі (Model Extraction)
Тисячі запитів до API → відтворення логіки моделі
🔓
Інверсія моделі (Model Inversion)
Витік навчальних даних через аналіз outputs моделі

Захист ML-моделей

🛡️ Adversarial training — навчання на adversarial examples для стійкості
✅ Input validation — перевірка вхідних даних на аномалії перед inference
📊 Drift detection — моніторинг зміни розподілу даних та performance моделі
🔒 Rate limiting — обмеження запитів до API моделі проти extraction
🔗 Ensemble methods — атака на одну модель не обходить весь ансамбль

Bias та Fairness

⚖️ Проблема — AI може дискримінувати за расою, статтю, віком, локацією
🏦 Credit scoring — біас проти меншин = порушення антидискримінаційних законів
🔍 Аудит — регулярна перевірка fairness метрик: demographic parity, equalized odds
📋 SR 11-7 — Federal Reserve: незалежна валідація моделей, документування, моніторинг

Приклади Adversarial-атак

💳
Ухилення у виявленні шахрайства (Evasion in Fraud Detection)
Розбити $10K на 47 транзакцій
Додати легітимні покупки між fraud
Імітувати нормальний час доби
Використати «довірений» merchant
🔍
Model Extraction через API
50K запитів до scoring API
Аналіз відповідей: score + причина
Побудова surrogate моделі
Тестування evasion на копії

Реальний кейс: злочинна група витратила 3 місяці на reverse engineering fraud-моделі банку через API, потім провела $2M fraud.

Техніки Model Poisoning

💉
Отруєння даних (Data Poisoning)
Впровадження шкідливих зразків у навчальні дані: fraud-транзакції маркуються як легітимні
🚪
Атака через бекдор (Backdoor Attack)
Тригер у input (конкретний merchant + час) = модель завжди пропускає fraud
🔧
Підробка моделі (Model Tampering)
Несанкціонована зміна ваг моделі у production через supply chain compromise
🛡️
Захист
Data validation, model signing, integrity monitoring, canary transactions

AI Governance Framework

1
Інвентаризація (Inventory)
Реєстр усіх AI/ML моделей: ціль, дані, ризик, відповідальний власник
2
Оцінка ризиків (Risk Assessment)
Класифікація моделей за рівнем ризику: low / medium / high / critical
3
Валідація (Validation)
Незалежна валідація: accuracy, fairness, robustness, explainability тести
4
Моніторинг (Monitoring)
Production моніторинг: drift detection, performance degradation, bias alerts
5
Журнал аудиту (Audit Trail)
Повний аудит: версії моделей, дані навчання, рішення, зміни для регулятора

EU AI Act: рівні ризику

🚫 Неприйнятний ризик (заборонено) — social scoring, маніпулятивний AI, масове біометричне стеження у реальному часі
🔴 Високий ризик — credit scoring, fraud detection, HR-системи. Вимоги: human oversight, XAI, аудит, документація
🟡 Обмежений ризик — чатботи, deepfake-генератори. Вимога: прозорість, маркування AI-контенту
🟢 Мінімальний ризик — спам-фільтри, рекомендації, ігри. Без обов'язкових вимог

Штрафи до 35M EUR або 7% глобального обороту. Фінансовий сектор: більшість AI-систем = high-risk категорія.

Ключові висновки

1️⃣ Adversarial attacks — evasion, poisoning, extraction загрожують ML-моделям

3️⃣ Bias audit — регулярна перевірка fairness, SR 11-7 стандарт
