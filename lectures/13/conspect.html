13.1 Паролі та управління акаунтами

Слабкі та повторювані паролі залишаються одним із найпоширеніших векторів компрометації. За даними Verizon DBIR, понад 80% зломів, пов'язаних з хакерством, використовують вкрадені або слабкі паролі. Середній користувач має 100+ онлайн-акаунтів, що робить ручне управління унікальними паролями неможливим.

Password Managers:

  • Генерація та зберігання унікальних складних паролів для кожного сервісу (Bitwarden, 1Password, KeePass).
  • Master password — єдиний пароль, який потрібно запам'ятати. Повинен бути довгим (16+ символів) та унікальним.
  • Автозаповнення запобігає введенню паролю на фішинговому сайті: менеджер перевіряє домен.

Multi-Factor Authentication (MFA):

  • SMS OTP — найслабший фактор: SIM-swap, SS7-перехоплення. Краще, ніж нічого, але не для критичних акаунтів.
  • TOTP (Google Authenticator, Authy) — одноразовий код, генерований додатком. Значно безпечніший за SMS.
  • FIDO2/WebAuthn (Passkeys) — криптографічна автентифікація прив'язана до пристрою та домену. Phishing-resistant за дизайном. Підтримується Google, Apple, Microsoft.
  • Hardware keys (YubiKey) — фізичний пристрій для FIDO2. Золотий стандарт для високоцінних акаунтів.

Breach monitoring — сервіси типу Have I Been Pwned перевіряють, чи ваші облікові дані з'явилися у відомих витоках. Password managers (1Password, Bitwarden) мають вбудовану перевірку Watchtower/Vault Health.

Ентропія пароля — як оцінити стійкість:

Ентропія пароля вимірюється в бітах і показує кількість можливих комбінацій. Формула: H = L × log₂(N), де L — довжина пароля, N — розмір алфавіту.

# Розрахунок ентропії пароля (Python)
import math

def password_entropy(length, charset_size):
    """
    Обчислює ентропію пароля в бітах.
    length       — кількість символів у паролі
    charset_size — кількість можливих символів (алфавіт)
    """
    if charset_size <= 0 or length <= 0:
        return 0
    entropy = length * math.log2(charset_size)
    return round(entropy, 2)

# Приклади наборів символів:

Перевірка витоку через Have I Been Pwned API (Python):

# Перевірка пароля через Have I Been Pwned API (k-Anonymity model)
# Пароль НІКОЛИ не передається на сервер — лише перші 5 символів SHA-1 хешу
import hashlib
import requests

def check_password_pwned(password):
    """
    Перевіряє, чи був пароль у відомих витоках даних.
    Використовує k-Anonymity: на сервер відправляється лише prefix хешу.
    """
    # 1. Хешуємо пароль SHA-1
    sha1_hash = hashlib.sha1(password.encode('utf-8')).hexdigest().upper()

    # 2. Розділяємо хеш: prefix (5 символів) та suffix (решта)
    prefix = sha1_hash[:5]
    suffix = sha1_hash[5:]

    # 3. Запитуємо API з prefix — отримуємо всі suffix-и з цим prefix
    url = f"https://api.pwnedpasswords.com/range/{prefix}"
    response = requests.get(url, headers={"Add-Padding": "true"})
    response.raise_for_status()

    # 4. Шукаємо наш suffix у відповіді
    for line in response.text.splitlines():
        hash_suffix, count = line.split(":")
        if hash_suffix.strip() == suffix:
            return int(count)
    return 0

# Приклад використання:
password = "P@ssw0rd123"
times_found = check_password_pwned(password)

if times_found > 0:
    print(f"УВАГА! Пароль знайдено у {times_found} витоках даних!")
    print("Негайно змініть цей пароль!")
else:
    print("Пароль не знайдено у відомих витоках.")

    if response.status_code == 200:
        breaches = response.json()
        print(f"Email знайдено у {len(breaches)} витоках:")
        for breach in breaches:
            print(f"  - {breach['Name']} ({breach['BreachDate']})")
    elif response.status_code == 404:
        print("Email не знайдено у відомих витоках.")

# Спрощена схема реєстрації та автентифікації з Passkey

# === РЕЄСТРАЦІЯ (Registration / Attestation) ===
#
# 1. Користувач → Сервер:  "Хочу зареєструвати passkey"
# 2. Сервер → Браузер:     Challenge (випадкові байти) + RelyingParty ID (домен)
# 3. Браузер → Автентифікатор: Запит на створення ключової пари
# 4. Автентифікатор:       Біометрія / PIN для підтвердження
# 5. Автентифікатор:       Генерує пару ключів (приватний + публічний)
#                          Приватний ключ ЗАЛИШАЄТЬСЯ на пристрої
# 6. Браузер → Сервер:     Публічний ключ + підписаний Challenge
# 7. Сервер:               Зберігає публічний ключ для користувача

# === АВТЕНТИФІКАЦІЯ (Authentication / Assertion) ===
#
# 1. Користувач → Сервер:  "Хочу увійти"
# 2. Сервер → Браузер:     Challenge + RelyingParty ID (домен!)
# 3. Браузер:              Перевіряє, що домен == RP ID (захист від фішингу!)
# 4. Автентифікатор:       Біометрія / PIN
# 5. Автентифікатор:       Підписує Challenge приватним ключем
# 6. Браузер → Сервер:     Підпис
# 7. Сервер:               Перевіряє підпис публічним ключем → Доступ надано

# Чому Passkeys стійкі до фішингу:
# - Приватний ключ НІКОЛИ не покидає пристрій
# - Браузер автоматично перевіряє домен (RP ID)
# - evil-bank.com ≠ bank.com → автентифікатор відмовить
# - Немає "секрету", який можна перехопити (на відміну від пароля чи OTP)

13.2 Захист персональних даних

Персональні дані є цінним активом для зловмисників: від identity theft до відкриття фінансових рахунків на чуже ім'я. Захист приватності — це не лише право, а й фінансова безпека.

GDPR та принципи захисту даних:

  • Data minimization — збирайте лише ті дані, які дійсно потрібні. Не зберігайте «про всяк випадок».
  • Purpose limitation — дані використовуються лише для мети, заради якої зібрані.
  • Storage limitation — видаляйте дані, коли вони більше не потрібні.
  • Право на забуття — клієнт має право вимагати видалення своїх персональних даних.
  • Data breach notification — повідомлення регулятора протягом 72 годин, постраждалих — «без невиправданої затримки».

Практичні кроки для приватності:

  • Обмежте публічну інформацію в соцмережах (LinkedIn, Facebook) — це OSINT для spear phishing.
  • Використовуйте різні email-адреси для різних категорій (фінанси, соцмережі, shopping).
  • Перевіряйте дозволи мобільних додатків: камера, мікрофон, локація — лише коли дійсно потрібно.
  • Використовуйте приватний режим браузера, блокувальники трекерів (uBlock Origin), DNS-фільтрацію.

13.3 Безпека пристроїв та мереж

Кінцеві пристрої (ноутбуки, смартфони, планшети) та домашні мережі є першою лінією оборони для більшості користувачів. Компрометація пристрою = доступ до банківських додатків, email, корпоративних систем.

Endpoint Security:

  • Оновлення — автоматичне оновлення ОС та додатків. 60% успішних атак використовують відомі, але невиправлені вразливості.
  • Антивірус / EDR — Windows Defender достатній для більшості, корпоративні рішення (CrowdStrike, SentinelOne) для бізнесу.
  • Full Disk Encryption — BitLocker (Windows), FileVault (macOS), LUKS (Linux). При крадіжці пристрою дані захищені.
  • Screen lock — автоматичне блокування через 1-5 хвилин, PIN/біометрія для розблокування.

Безпека мережі:

  • Домашній Wi-Fi — WPA3 (або WPA2), складний пароль, зміна default admin credentials роутера, оновлення firmware.
  • Публічний Wi-Fi — НІКОЛИ не вводьте фінансові дані у публічних мережах без VPN. MITM-атаки тривіальні в кафе та аеропортах.
  • VPN — шифрований тунель для публічних мереж. Надійні провайдери: WireGuard-based рішення, корпоративний VPN для роботи.
  • IoT — розумні пристрої (камери, термостати) в окрему мережу (VLAN/guest network), регулярні оновлення firmware.

Приклад правил брандмауера (iptables / nftables):

# Базові правила брандмауера для Linux-сервера (iptables)
# Демонструє принцип "заборонити все, дозволити необхідне"
# ========================================================

#!/bin/bash
# firewall-setup.sh

# Очистити існуючі правила
iptables -F
iptables -X

# Політика за замовчуванням: ЗАБОРОНИТИ все
iptables -P INPUT DROP       # Вхідний трафік — заборонено
iptables -P FORWARD DROP     # Транзитний трафік — заборонено
iptables -P OUTPUT ACCEPT    # Вихідний трафік — дозволено

# Дозволити loopback (localhost)
iptables -A INPUT -i lo -j ACCEPT

# Дозволити вже встановлені з'єднання (stateful firewall)
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Дозволити SSH (тільки з довіреної мережі)
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT

# Дозволити HTTPS (для веб-сервера)
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Дозволити DNS відповіді
iptables -A INPUT -p udp --sport 53 -j ACCEPT

# Захист від ping flood (обмеження ICMP)
iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s -j ACCEPT

# Блокувати та логувати все інше
iptables -A INPUT -j LOG --log-prefix "FIREWALL-BLOCKED: "
iptables -A INPUT -j DROP

# Зберегти правила
# Debian/Ubuntu: iptables-save > /etc/iptables/rules.v4
# CentOS/RHEL:   service iptables save

13.4 Фінансова безпека онлайн

Онлайн-фінанси вимагають підвищеної пильності: від безпечних платежів до розпізнавання інвестиційних скамів.

Безпечні онлайн-платежі:

  • Використовуйте віртуальні картки для онлайн-покупок (одноразові номери, ліміти).
  • Перевіряйте HTTPS та легітимність сайту перед введенням даних картки.
  • Apple Pay / Google Pay безпечніші за введення номера картки (токенізація).
  • Увімкніть push-сповіщення про транзакції для миттєвого виявлення несанкціонованих операцій.

Розпізнавання шахрайства:

  • Investment scams — обіцянки гарантованого прибутку 100%+, «ексклюзивна можливість», тиск на швидке рішення. Якщо звучить занадто добре — це скам.
  • Romance scams — побудова онлайн-стосунків із подальшим запитом на гроші. Особливо поширені на dating-платформах.
  • Fake apps — підроблені банківські додатки в App Store / Google Play. Завантажуйте лише з офіційних посилань банку.
  • Правило: банк НІКОЛИ не просить PIN, CVV, повний номер картки або OTP-код по телефону або email.

Моніторинг та реагування: підключіть SMS/push-сповіщення про всі операції, регулярно перевіряйте виписки, використовуйте сервіси моніторингу кредитної історії, негайно блокуйте картку при підозрілій транзакції через мобільний додаток.

Складна або незрозуміла схема "Арбітраж AI-алгоритмів на блокчейні" Якщо не можете пояснити, як саме заробляються гроші — не інвестуйте.

Перевірка SSL-сертифіката сайту (Python):

# Перевірка SSL-сертифіката вебсайту за допомогою Python
# ======================================================
import ssl
import socket
from datetime import datetime

def check_ssl_certificate(hostname, port=443):
    """
    Отримує та аналізує SSL-сертифікат вебсайту.
    Корисно для перевірки легітимності фінансового сайту.
    """
    context = ssl.create_default_context()

    try:
        with socket.create_connection((hostname, port), timeout=10) as sock:
            with context.wrap_socket(sock, server_hostname=hostname) as ssock:
                cert = ssock.getpeercert()

                # Основна інформація
                print(f"{'='*50}")
                print(f"SSL-сертифікат для: {hostname}")
                print(f"{'='*50}")

                # Кому виданий (Subject)
                subject = dict(x[0] for x in cert['subject'])
                print(f"Організація:    {subject.get('organizationName', 'Не вказано')}")
                print(f"Домен (CN):     {subject.get('commonName', 'Не вказано')}")

                # Хто видав (Issuer)
                issuer = dict(x[0] for x in cert['issuer'])
                print(f"Видавець:       {issuer.get('organizationName', 'Невідомо')}")
                print(f"Тип видавця:    {issuer.get('commonName', 'Невідомо')}")

                # Терміни дії
                not_before = datetime.strptime(cert['notBefore'], '%b %d %H:%M:%S %Y %Z')
                not_after = datetime.strptime(cert['notAfter'], '%b %d %H:%M:%S %Y %Z')
                days_left = (not_after - datetime.utcnow()).days

                print(f"Дійсний з:      {not_before.strftime('%Y-%m-%d')}")
                print(f"Дійсний до:     {not_after.strftime('%Y-%m-%d')}")
                print(f"Днів залишилось: {days_left}")

                # Alternative Names (які домени покриває)
                san = cert.get('subjectAltName', [])
                domains = [entry[1] for entry in san if entry[0] == 'DNS']
                print(f"Домени (SAN):   {', '.join(domains[:5])}")
                if len(domains) > 5:
                    print(f"                ... та ще {len(domains)-5} доменів")

                # Попередження
                print(f"\n--- Аналіз ---")
                if days_left < 30:
                    print(f"⚠ УВАГА: Сертифікат закінчується через {days_left} днів!")
                if subject.get('organizationName') is None:
                    print("⚠ DV-сертифікат (Domain Validation) — організація не перевірена")
                    print("  Це НЕ означає, що сайт шахрайський, але EV/OV надійніший")
                else:
                    print("✓ OV/EV-сертифікат — організація перевірена центром сертифікації")

                return cert

    except ssl.SSLCertVerificationError as e:
        print(f"ПОМИЛКА SSL: Сертифікат недійсний! {e}")
        print("НЕ ВВОДІТЬ жодних даних на цьому сайті!")
    except socket.timeout:
        print(f"Сайт {hostname} не відповідає.")
    except Exception as e:
        print(f"Помилка: {e}")

# Приклад використання:
check_ssl_certificate("privatbank.ua")

Підсумок

Кібергігієна — це щоденні практики, що значно зменшують ризик кібератак. Password manager + MFA (бажано FIDO2) закривають найпоширеніший вектор — компрометацію облікових даних. Захист персональних даних та обмеження цифрового сліду ускладнюють spear phishing та identity theft.

Оновлення пристроїв, шифрування дисків та обережність у публічних мережах захищають від технічних атак. Для фінансової безпеки: віртуальні картки, push-сповіщення та здоровий скептицизм до «вигідних пропозицій» є обов'язковими.

У наступній лекції ми розглянемо ризик-менеджмент та SOC: управління кіберризиками, побудову та функціонування центрів безпеки (Security Operations Center) у фінансових установах.
